`gravity`는 모바일 퍼스트 커뮤니티 웹앱을 만들려던 프로젝트였다. React로 화면을 만들고, Spring Boot로 API를 세우고, PostgreSQL과 Redis, Docker Compose까지 묶어서 인증부터 게시글까지 이어보는 것이 처음 목표였다.

그런데 실제로 남은 기억은 앱보다 개발 방식에 가깝다. 2026년 3월 24일 초기 골격을 만들고, 3월 26일부터 Claude Code 안에서 역할별 에이전트를 나누기 시작했다. 계획과 검토는 더 강한 모델이 맡고, 구현은 빠른 모델이 맡는 구조였다. 지금 보면 꽤 당연한 분업처럼 보이지만, 당시에는 "AI와 함께 프로젝트를 굴린다면 어느 지점에서 사람의 판단을 넣어야 하는가"를 몸으로 확인한 실험이었다.

## 왜 풀스택 커뮤니티였나

도메인은 일부러 흔한 커뮤니티로 잡았다. 게시글, 댓글, 추천, 로그인, 프로필 같은 기능은 누구나 이해한다. 도메인 설명에 에너지를 쓰기보다 프론트엔드, 백엔드, 인프라, 검증 루프가 어떻게 맞물리는지 보는 데 집중하고 싶었다.

프론트는 React, TypeScript, Vite, MUI, Zustand, TanStack Query를 골랐다. 인증 상태는 Zustand로 들고, 서버 상태는 TanStack Query로 분리하는 구조가 자연스러웠다. 백엔드는 Spring Boot, JPA, Spring Security, JWT를 썼다. 기존 업무 경험상 오래된 구조를 많이 봐왔기 때문에, 새 프로젝트에서는 요즘 쓰는 백엔드 스택을 처음부터 끝까지 잡아보고 싶었다.

인프라는 PostgreSQL, Redis, MinIO, Prometheus, Grafana, Loki까지 Docker Compose로 묶는 방향이었다. 지금 돌아보면 첫 버전치고는 넓었다. 하지만 이 프로젝트의 목적이 "게시판 하나를 빨리 만들기"보다 "풀스택 운영 흐름을 한 번에 만져보기"였기 때문에 선택 자체는 이해된다.

재미있는 점은 Java 버전이었다. 이상적으로는 Java 21을 기준으로 잡는 편이 안정적이었지만, 로컬에는 Java 25가 있었다. 그래서 Gradle toolchain과 플러그인 호환을 맞추는 쪽으로 흘렀다. 기술 선택이 항상 이상적인 설계에서 출발하지 않는다는 사실을 그대로 보여준 장면이었다.

## Phase 1 인증이 남긴 것

3월 26일 밤부터 3월 27일까지 가장 집중한 것은 인증이었다. 회원가입, 로그인, JWT 발급, refresh token 저장, 보호 API, 라우터 가드, 프론트 auth store, axios interceptor까지 이어지는 흐름을 만들었다.

구조는 단순히 "로그인 버튼을 누르면 토큰을 받는다"가 아니었다. access token은 짧게 유지하고, refresh token은 DB에 저장한다. 프론트에서는 사용자 정보만 지속 저장하고 access token은 메모리에 두는 쪽으로 잡았다. axios interceptor는 API 요청 중 인증 만료를 만나면 갱신을 시도한다. 이때 store와 axios 모듈이 서로를 직접 import하면 순환 참조가 생겨서, 전역 bridge를 두는 식으로 우회했다.

```txt
회원가입
  -> 로그인
  -> access token 발급
  -> 보호 API 호출
  -> 만료 시 refresh 시도
  -> 실패하면 로그아웃
```

여기서 Zustand와 TanStack Query를 나눈 이유도 있었다. "내가 지금 로그인한 사람인가"는 앱 전체의 클라이언트 상태에 가깝고, "게시글 목록과 프로필 데이터를 언제 다시 가져올 것인가"는 서버 캐시 문제에 가깝다. 둘을 한 store에 넣으면 편해 보이지만, 캐시 무효화와 인증 흐름이 금방 엉킨다.

다만 refresh token을 DB에 저장했다고 자동으로 안전해지는 것은 아니다. rotation, revoke, 만료 시간, 탈취 대응, XSS/CSRF 경계가 같이 설계되어야 한다. 이 프로젝트에서는 그 전체를 완성했다기보다, 첫 인증 루프에서 어떤 질문들이 생기는지 확인한 쪽에 가깝다.

큰 구조보다 더 오래 남은 것은 작은 UX였다. 처음에는 이메일이나 닉네임 중복이 409 응답으로만 처리됐다. 화면 상단에 에러가 뜨면 기능적으로는 맞지만 사용자는 어느 필드가 문제인지 바로 알기 어렵다. 그래서 응답의 error code를 읽어 해당 필드 아래에 메시지를 붙이고, 사용자가 값을 바꾸면 에러가 사라지도록 고쳤다. blur 시 중복 확인도 붙였다.

인증에서 가장 중요한 것은 JWT 흐름 같은 큰 단어가 아니라, 실패했을 때 사용자가 다음 행동을 바로 알 수 있는가였다. AI는 큰 설계를 빨리 만들었지만 이런 "제품성의 작은 단위"는 한 번에 잡지 못했다.

## 검증은 어떻게 했나

문서에는 `make dev`, `make web`, `make api`, `make lint`, `make verify` 같은 진입점을 두었다. 실제 로그에서는 상황에 따라 더 직접적인 명령도 반복했다.

```bash
cd /path/to/gravity/apps/api && ./gradlew compileJava
cd /path/to/gravity/apps/api && ./gradlew bootRun --args='--spring.profiles.active=dev'
cd /path/to/gravity/apps/web && npx tsc --noEmit
cd /path/to/gravity/apps/web && npx eslint src/ --max-warnings 0
```

이 검증은 완성 선언을 위한 의식이 아니라, AI가 만든 변경이 실제 로컬 환경에서 돌아가는지 확인하는 최소 장치였다. 특히 백엔드와 프론트가 동시에 바뀌는 프로젝트에서는 한쪽만 성공해도 전체 기능이 깨질 수 있다. 타입 체크, 컴파일, 린트, API 기동 확인을 나누어 본 이유가 거기에 있다.

## 오래 걸린 문제들

삽질은 예상보다 낮은 곳에서 시작됐다. Gradle wrapper가 없거나, Java toolchain이 맞지 않거나, Gradle과 foojay 플러그인 버전이 충돌했다. Spring Boot와 문서화 라이브러리 버전이 맞지 않는 문제도 있었다. DB 연결은 IPv6 localhost 때문에 흔들렸고, 프론트에서는 Zustand selector가 무한 렌더링을 만들었다. Vite 타입 import 문제도 자주 나왔다.

이런 문제들은 설계 문서에서는 거의 보이지 않는다. 그런데 실제 개발 시간은 이런 호환성, 로컬 환경, 빌드 도구, 브라우저 런타임에서 많이 사라진다. AI는 "이론적으로 맞는 구조"를 빨리 내놓지만, 현재 머신에 어떤 JDK가 있고, 어떤 wrapper가 없고, 어떤 플러그인이 충돌하는지는 결국 명령을 실행해봐야 드러난다.

## 에이전트 운영이 과했다

이 프로젝트의 가장 큰 시행착오는 앱보다 운영 규칙이 먼저 커졌다는 점이다. `CLAUDE.md`에는 phase 표, 역할별 에이전트, Opus 에스컬레이션 조건, Git 전략, 금지 패턴까지 들어갔다. Orchestrator, Implementer, Reviewer, Verifier, Tester, Explorer, Tech Advisor 같은 역할도 나누었다.

문서가 전부 나쁘지는 않았다. 이후 작업에서 "계획은 강한 모델, 반복 구현은 빠른 모델, 불확실한 판단은 다시 상위 모델"이라는 기준은 계속 재사용됐다. 하지만 Phase 2 게시글 CRUD는 구현보다 설계 문서와 계획서가 먼저 커졌다. 기능이 적은 시점에는 무거운 운영 체계가 프로젝트를 앞으로 밀기보다 붙잡을 수 있다.

AI도 이 방향을 더 키우는 경향이 있었다. 내가 원하는 것은 "이메일 중복 UX를 더 자연스럽게"였는데, AI는 검증 체계와 문서 구조를 넓히려 했다. 반대로 실제 화면에서 사용자에게 어색한 피드백은 사람의 지적이 들어간 뒤에야 좁혀졌다.

## 지금 다시 한다면

다시 한다면 Phase 0과 Phase 1의 범위를 더 작게 잡을 것이다. "인증 + 게시글 하나가 실제로 보이는 화면"까지를 첫 완료 기준으로 두고, 에이전트 운영 문서는 그 뒤에 키우는 편이 낫다.

그래도 `gravity`는 실패한 프로젝트가 아니다. 기능보다 더 중요한 습관이 생겼다. 큰 판단은 모델을 올리고, 구현은 작게 나누고, 검증 명령은 실제로 돌리고, 사용자 경험은 화면에서 다시 확인한다. 이후 작업에서 내가 AI에게 계속 요구하게 된 기준은 여기서 꽤 많이 만들어졌다.
