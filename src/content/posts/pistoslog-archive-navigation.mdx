블로그를 만들 때 메뉴를 많이 넣으면 처음에는 풍성해 보인다. Writing, Notes, Projects, About, Now가 모두 있으면 기록 체계가 잘 갖춰진 것처럼 보인다. 하지만 실제로 글을 쌓기 시작하면 질문이 바뀐다. 이 메뉴가 정말 독자에게 필요한가. 내가 계속 운영할 수 있는 구조인가.

이번 정리는 그 질문에서 시작했다. `Notes`와 `Now`는 지금의 `pistoslog`에 필요하지 않았다. 짧은 메모는 개인 위키 쪽이 더 맞고, 지금 하고 있는 일은 글과 프로젝트 안에 자연스럽게 녹이는 편이 낫다. 그래서 블로그에서는 완성된 글과 작업물에 집중하기로 했다.

## 문제는 태그와 카테고리가 장식이었다는 점

홈에는 태그 모아보기가 있었다. 하지만 태그는 `span`으로 렌더링되어 있어서 눌러도 아무 일도 일어나지 않았다. 화면에는 `#Astro`, `#회고`, `#LLM Wiki` 같은 단어가 보이지만, 실제 탐색 기능은 없었다.

Writing의 카테고리 카드도 비슷했다. `일상`, `생각`, `기술공부`, `작업기록` 카드가 있었지만 링크는 `/writing/#tech` 같은 앵커였다. 클릭하면 전체 글 목록 안의 특정 섹션으로 스크롤할 뿐, “기술공부 글만 보기” 같은 명확한 상태가 생기지 않았다.

이건 블로그가 작을 때는 괜찮아 보인다. 하지만 글이 늘어나면 앵커 이동은 곧 한계가 온다. 사용자는 카테고리를 눌렀을 때 해당 글만 보고 싶어 한다. 태그를 눌렀을 때도 그 태그가 달린 글 목록으로 이동하길 기대한다.

바꾸기 전과 후를 놓고 보면 더 분명하다.

| 이전 | 이후 |
| --- | --- |
| 태그는 `span`이라 눌러도 이동하지 않는다 | 태그는 `/tags/[tag]/` 글 목록으로 이동한다 |
| 카테고리는 `/writing/#tech` 앵커로 스크롤한다 | 카테고리는 `/writing/tech/` 전용 목록으로 이동한다 |
| URL이 상태를 설명하지 못한다 | URL만 봐도 현재 archive 범위를 알 수 있다 |

## 정적 아카이브 URL로 바꿨다

구조는 단순하게 잡았다.

```txt
/writing/
/writing/[category]/
/writing/[category]/tags/[tag]/
/tags/
/tags/[tag]/
```

`/writing/`은 전체 글 목록이다. 카테고리 카드를 누르면 `/writing/tech/`처럼 전용 페이지로 이동한다. `/tags/`는 전체 태그 인덱스이고, 각 태그는 `/tags/회고/`, `/tags/llm-wiki/` 같은 정적 페이지가 된다. 카테고리 안에서 누른 태그는 `/writing/tech/tags/python/`처럼 카테고리 맥락을 유지한 글 목록으로 보낸다.

Astro에서는 이 방식이 자연스럽다. content collection으로 글을 모으고, `getStaticPaths()`에서 필요한 경로를 빌드 시점에 생성하면 된다. [Astro 라우팅 문서](https://docs.astro.build/en/guides/routing/#static-ssg-mode)도 static mode의 dynamic route는 빌드 시점에 생성할 `params` 목록을 `getStaticPaths()`로 돌려줘야 한다고 설명한다. 서버 상태나 클라이언트 필터가 없어도 공유 가능한 URL이 생기고, GitHub Pages 같은 정적 배포에도 잘 맞는다.

태그는 조금 더 조심했다. 한글, 공백, 대소문자, 영문 약어가 섞인다. 그래서 화면에 보이는 label과 URL에 쓰는 slug를 분리했다. `LLM Wiki`는 `/tags/llm-wiki/`가 되고, `회고`는 한글 slug를 그대로 URL encoding해서 쓴다.

추가로 서로 다른 태그가 같은 slug로 접히면 빌드에서 실패하게 했다. 예를 들어 미래에 `C++`와 `C#` 같은 태그가 같은 문자열로 정규화되면 조용히 합쳐지지 않아야 한다. 글 탐색에서 조용한 병합은 나중에 찾기 어려운 버그가 된다.

검사는 단순하다. 화면에 보이는 label을 정규화한 값과 URL slug를 따로 만들고, 같은 slug인데 정규화 label이 다르면 에러를 던진다.

```ts
if (existing.normalizedLabel !== normalizedLabel) {
  throw new Error(`Tag slug collision: "${existing.label}" and "${label}" both map to "${slug}".`);
}
```

## Writing 안에도 태그를 다시 배치했다

홈의 태그 모아보기는 전체 블로그의 흐름을 보여준다. 하지만 Writing 카테고리 페이지에서는 범위가 달라야 한다. `기술공부` 페이지에 들어갔으면 기술 글에 등장한 태그만 보는 편이 맞다. `작업기록` 페이지에서는 작업 글의 태그만 모아야 한다.

그래서 Writing 카테고리 카드 아래에 작은 태그 영역을 추가했다. `/writing/`에서는 전체 글 기준 태그를 보여주고, `/writing/work/`에서는 작업기록 글에 등장한 태그만 보여준다. 태그를 누르면 전역 태그 아카이브가 아니라 해당 카테고리 안에서 그 태그가 달린 글 목록으로 이동한다.

처음에는 전역 태그 아카이브만 써도 충분하다고 봤다. 하지만 `기술공부`에서 `#회고`를 눌렀는데 작업기록 글까지 섞이면 사용자가 방금 선택한 범위가 풀려버린다. 그래서 category-scoped tag URL을 따로 만들었다. URL은 조금 길어졌지만, 탐색 상태는 훨씬 분명해졌다.

## Notes와 Now는 완전히 제거했다

메뉴에서만 숨기는 것으로 끝내지 않았다. `notes` collection, `/notes/` 라우트, `/now/` 페이지, `new:note` 스크립트, README의 노트 작성 안내까지 같이 제거했다.

이런 정리는 사소해 보이지만 중요하다. 남아 있는 생성 명령이나 content schema는 다음 작업자를 헷갈리게 만든다. “화면에는 없지만 구조에는 남아 있는 기능”은 시간이 지나면 다시 살아나거나, 빌드 오류의 원인이 된다.

기존 글 본문에도 `posts`, `notes`, `projects`로 나뉜다고 설명한 문장이 있어서 현재 구조에 맞게 고쳤다. 이제 블로그는 `posts`와 `projects` 두 축으로 설명된다.

## 검증과 리뷰

구현 후에는 `npm run check`, `npm run build`, `npm run verify:architecture`, `git diff --check`를 돌렸다. 빌드 결과에서 `/writing/thought/`, `/writing/tech/`, `/writing/work/`, `/writing/tech/tags/python/`, `/tags/회고/`, `/tags/llm-wiki/` 같은 페이지가 정적으로 생성되는 것도 확인했다.

Browser Use로 실제 클릭 흐름도 봤다. 카테고리에서 `/writing/work/`로 이동하는지, 카테고리 태그가 `/writing/tech/tags/python/`처럼 범위를 유지하는지, 글 상세의 `#회고` 태그가 `/tags/회고/`로 이동하는지, `/notes/`와 `/now/`가 더 이상 블로그의 정상 페이지로 남아 있지 않은지 확인했다.

서브에이전트 리뷰에서는 태그 slug 충돌 위험이 나왔다. 현재 콘텐츠에서는 문제가 없었지만, 미래에 다른 label이 같은 slug를 만들 수 있다는 지적이었다. 이 피드백을 반영해 충돌이 생기면 빌드 시점에 에러를 던지도록 바꿨다.

이번 작업의 결론은 단순하다. 눌러도 아무 일 없는 태그는 그냥 장식이었다. 태그는 눌렀을 때 태그 글 목록으로 가야 하고, 카테고리는 해당 카테고리 글 목록으로 가야 한다. 메뉴는 운영할 수 있는 것만 남겨야 한다. 작아 보이는 정리였는데, 막상 덜어내고 연결하니 블로그가 덜 허세스러워졌다.
