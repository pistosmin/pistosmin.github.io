좋은 AI 도구가 있어도 업무 환경에서는 그대로 쓰기 어렵다. 웹 검색이 막히고, 패키지 설치가 제한되고, 외부 플러그인 마켓플레이스에 접근할 수 없고, 허용된 경로로만 LLM을 호출해야 할 수 있다. 로컬에서는 당연했던 개발 루프도 제한된 환경에서는 하나씩 확인해야 할 조건이 된다.

2026년 4월 17일에 했던 작업은 이 제약을 정면으로 본 실험이었다. 하나는 제한된 실행 환경에서 허용된 AI 호출 경로를 CLI처럼 감싸는 핸드오프 패키지였고, 다른 하나는 같은 작업을 여러 번 반복하는 Ralph 스타일 루프를 오프라인 스킬로 만드는 일이었다.

둘 다 완성된 자동화라기보다는 "제약된 환경에 가져갈 수 있는 형태로 AI 작업을 포장하는 법"에 가까웠다.

## 왜 MCP가 아니라 CLI였나

처음 떠올리기 쉬운 답은 MCP 서버다. 도구를 서버로 만들고, Claude나 다른 클라이언트가 호출하게 하면 구조가 깔끔해 보인다. 하지만 업무 환경에서는 깔끔함보다 반입 가능성과 운영 부담이 먼저다. 상시 로컬 서버를 띄워야 하고, 포트와 권한을 관리해야 하고, 설치 정책을 통과해야 한다.

그래서 첫 선택은 Python CLI였다. 모델은 자연어를 이해하되, 실제 실행은 정해진 명령으로 제한한다. 질문을 보내고, 응답을 받고, 로컬 히스토리를 남기고, 종료 코드를 돌려주는 결정론적 부분은 Python이 맡는다. Claude 스킬은 얇은 트리거 역할만 한다.

이 구조는 멋은 덜하지만 설명하기 쉽다. 대상 PC에 파일을 풀고, 설정을 넣고, 명령을 실행한다. 실패하면 로그를 본다. AI 도구라기보다 작은 운영 도구에 가깝다.

당시 기준으로는 이렇게 봤다.

| 방식 | 장점 | 부담 |
| --- | --- | --- |
| MCP 서버 | 도구 인터페이스가 깔끔하고 확장성이 좋다 | 상시 프로세스, 포트, 설치 정책, 클라이언트 연동을 같이 봐야 한다 |
| Python CLI | 파일로 옮기기 쉽고 로그와 종료 코드를 통제하기 좋다 | 대화형 제품처럼 보이진 않고 사용성이 투박하다 |
| zip 패키지 | 네트워크 없이 전달하고 해시로 확인하기 좋다 | 업데이트와 버전 관리는 사람이 더 신경 써야 한다 |

제한된 환경에서는 "가장 예쁜 아키텍처"보다 "설치 설명이 짧고 실패 로그가 남는 방식"이 더 현실적이었다.

## 약한 모델에게 일을 넘기는 법

핸드오프 패키지에서 가장 중요했던 것은 "다음 사람이 강한 모델을 쓰지 못할 수 있다"는 전제였다. 그래서 거대한 설명 하나보다 작은 phase 문서가 필요했다. 무엇을 먼저 만들고, 어떤 파일을 고치고, 어떤 smoke checklist를 통과해야 하는지 나눠야 했다.

개인용 패키지와 팀용 패키지를 분리한 것도 같은 이유였다. 개인용에는 로컬 설정이 들어갈 수 있지만, 팀용에는 민감 설정이 없어야 한다. zip과 해시, 설치 스크립트, README를 한 묶음으로 만들면 외부 네트워크가 없는 환경에서도 전달과 검증이 가능하다.

이때 공개 저장소는 특히 조심해야 했다. 비공개 호스트명, 인증 흐름, 실제 계정 형태, 조직명은 그대로 남기지 않는다. "제한된 네트워크에서만 접근 가능한 웹 기반 AI 도구", "조직 인증 흐름", "제한된 모델 권한을 가진 개발 환경"처럼 구조는 남기되 식별자는 지운다.

## Ralph 루프는 왜 매번 새 세션으로 돌렸나

두 번째 작업은 Ralph 스타일 반복 루프였다. 공식적인 방식은 현재 세션을 계속 붙잡고 같은 프롬프트를 재주입하는 쪽에 가깝다. 내가 만든 버전은 다르게 잡았다. 부모 세션 또는 bash orchestrator가 `PROMPT.md`를 고정 입력으로 두고, 매 iteration마다 새 child session을 띄운다.

```bash
claude -p -- "$(cat PROMPT.md)"
```

child 출력은 현재 대화 컨텍스트에 계속 붙이지 않고 로그 파일에 누적한다. 이렇게 하면 이전 대화의 잡음이 다음 시도에 덜 섞인다. 장기적인 설계 판단에는 부적합할 수 있지만, 반복 가능한 refactor, 테스트 보강, 문서 정리처럼 목표가 명확한 작업에는 fresh context가 오히려 안정적일 수 있다.

```txt
parent orchestrator
  -> child session 1 -> log/verify
  -> child session 2 -> log/verify
  -> child session 3 -> DONE.md or stop condition
```

fresh context가 항상 좋은 것은 아니다. 긴 설계 맥락, 제품 판단, 사용자 피드백 누적이 중요한 작업에서는 오히려 손해다. 하지만 "이 체크리스트를 반복해서 통과시켜라"처럼 입력과 완료 조건이 좁은 작업에서는 이전 시도의 잡음이 덜 섞이는 장점이 있었다.

대신 멈춤 조건이 반드시 필요했다. 그래서 `DONE.md`와 검증 명령, 최대 반복 횟수, 최대 실행 시간, 진행 없음 감지, 연속 오류 중단 조건을 넣었다. 무한히 "조금 더 고치기"를 반복하는 AI 루프는 자동화가 아니라 사고다.

## 검증한 것과 못 한 것

Ralph 쪽은 mock Claude로 orchestration 로직을 테스트했다. 실제 LLM을 계속 호출하는 대신 iteration별 출력, 종료 코드, side effect를 흉내 내는 스크립트를 두고, stop condition이 맞게 동작하는지 확인했다.

검증 범위는 명확했다.

- `max_iter`에서 멈추는가
- `max_wall`에서 멈추는가
- 진행이 없으면 멈추는가
- 연속 오류를 세는가
- `DONE.md`와 `verify_cmd`를 함께 보는가
- zip 패키징과 sha256 생성이 맞는가

반대로 검증하지 못한 것도 분명했다. mock은 진짜 Claude의 장기 판단 품질, 비용, 토큰 소모, 위험한 편집, 제한된 환경의 권한 문제를 확인하지 못한다. CLI 쪽도 로컬 Mac에서는 실제 대상 환경에 접근할 수 없었기 때문에, 저장소의 테스트 문서는 자동 E2E라기보다 대상 환경에서 수행할 smoke checklist에 가까웠다.

이 구분을 글에 남기는 것이 중요하다. "오프라인 자동화 완성"이라고 쓰면 과장이다. 정확히는 "반입 가능한 구조와 검증 가능한 루프 조건을 준비했다"가 맞다.

## 오래 걸린 실수들

제약 환경을 과소평가한 순간이 많았다. 처음에는 문서를 런타임에 가져오면 될 것처럼 생각하기 쉽다. 하지만 제한된 네트워크에서는 WebFetch도, public npm도, 플러그인 마켓플레이스도 없을 수 있다. 그래서 필요한 reference는 빌드 단계에서 패키지 안에 굽는 쪽으로 바꿨다.

Ralph 구현에서는 `verify_cmd`의 exit code가 `tee` 파이프에 묻히는 문제가 있었다. 실제 검증 명령이 실패했는데 파이프의 마지막 명령만 보고 성공으로 처리될 수 있었다. 이건 `PIPESTATUS[0]`를 확인하는 방식으로 고쳤다. zip의 sha256 경로가 잘못 잡혀 packaging test가 실패한 문제도 있었다.

```bash
verify_cmd | tee verify.log
status=${PIPESTATUS[0]}
exit "$status"
```

공개화도 생각보다 까다로웠다. 파일 내용만 마스킹하면 끝이 아니었다. working tree, zip 산출물, dist 폴더, `.DS_Store`, 커밋 메타데이터까지 확인해야 한다. AI는 "민감한 문자열 몇 개 바꾸기"를 공개 준비로 착각하기 쉽다. 실제로는 산출물과 이력 전체를 보는 일에 가깝다.

## 가져갈 수 있는 체크리스트

업무 환경용 AI 자동화에서 먼저 확인할 것은 모델 이름이 아니었다.

- 대상 PC에서 실행 가능한 명령인가.
- 네트워크 없이도 필요한 reference를 읽을 수 있는가.
- 실패 로그가 남는가.
- zip과 해시로 전달물을 확인할 수 있는가.
- 검증 명령의 exit code를 제대로 보존하는가.
- 무한 반복 전에 멈출 조건이 있는가.

화려한 agent 아키텍처보다 반입 가능한 zip, 확인 가능한 해시, 읽을 수 있는 로그, 멈출 수 있는 루프가 먼저다. 제한된 환경에서의 AI 루프는 "똑똑한 대화"라기보다 운반 가능하고 검증 가능한 작은 제품에 가깝다.
