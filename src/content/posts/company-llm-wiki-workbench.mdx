레거시 시스템은 한 화면만 봐서는 이해되지 않는다. 예를 들어 화면에 표시된 숫자 하나가 왜 틀렸는지 보려면, 화면 이벤트에서 시작해 facade, session, DAO, SQL, 저장 프로시저와 테이블까지 따라가야 한다. 실제 장애나 데이터 문의는 대부분 이 체인을 거슬러 올라가야 풀린다.

문제는 사람이 이 체인을 매번 처음부터 다시 읽는다는 점이다. 문서가 있어도 오래됐고, 소스는 넓고, DB 오브젝트는 별도 세계처럼 떨어져 있다. 그래서 생각한 방식은 한 번에 모든 시스템을 문서화하는 것이 아니었다. LLM이 객체 하나를 읽고, 노트 하나를 만들고, 링크를 하나씩 쌓는 방식이었다.

이 작업은 실제 업무 데이터를 집으로 가져오는 프로젝트가 아니었다. 집에서는 템플릿, 프롬프트, 스킬 명세, 가짜 샌드박스만 만들고, 실제 분석은 업무 환경에서만 수행한다. 이 경계가 프로젝트의 첫 번째 설계였다.

## 보안 경계가 설계보다 먼저였다

편한 길은 항상 위험하다. 실제 소스 한두 개를 복사해오면 프롬프트를 만들기 쉽다. 실제 테이블명을 넣으면 예시가 더 그럴듯해진다. 하지만 그 순간 개인 프로젝트와 업무 자료의 경계가 무너진다.

그래서 워크벤치 repo에는 실데이터를 두지 않는 기준을 먼저 세웠다. 집에서 만든 것은 업무 환경으로 가져갈 수 있지만, 업무 환경의 원천은 집으로 가져오지 않는다. 실제 소스와 DB 오브젝트는 업무 환경의 형제 폴더에 read-only로 두고, LLM은 `llm-wiki` 영역에만 write한다.

이 경계는 단순한 보안 문구가 아니라 실행 구조다. 프롬프트도 "어디를 읽고 어디에 써야 하는가"를 명확히 한다. 개인 도구와 업무 도구를 섞지 않고, 공개 가능한 회고에서는 내부명과 실제 객체명을 일반화한다.

## 템플릿을 먼저 만든 이유

LLM에게 "이 시스템을 분석해줘"라고 하면 결과는 매번 달라진다. 어떤 날은 요약을 길게 쓰고, 어떤 날은 링크를 빼먹고, 어떤 날은 모르는 것을 아는 것처럼 쓴다. 그래서 타입별 템플릿을 먼저 만들었다.

대상은 procedure, function, package, table, view, query, facade, session, dao, screen 같은 10개 타입이었다. 각 타입은 frontmatter와 본문 구조가 다르지만, 공통적으로 원천 경로, 원천 해시, 분석 날짜, 호출 관계, 읽는 대상, 쓰는 대상, 모르는 점을 남긴다.

핵심 필드는 `source_path`, `source_hash`, `last_analyzed`, `calls`, `reads`, `writes`, `unknowns`였다.

`source_hash`는 특히 중요했다. 노트가 어떤 소스 버전을 보고 작성됐는지 알아야 한다. 해시가 같으면 확정된 내용을 다시 쓰지 않고, 해시가 달라졌을 때 전면 재검증한다. `last_analyzed`는 운영적으로 언제 봤는지 알려준다. `unknowns`는 LLM이 모르는 것을 모른다고 남기게 하는 장치다.

## 한 사이클 한 노트

작업 단위도 작게 잡았다. 한 번의 LLM 사이클은 노트 하나만 만든다. 한 번에 여러 객체를 분석하면 링크 품질, 큐 상태, 실패 복구가 꼬인다. 어느 객체에서 실패했는지도 흐려진다.

흐름은 대략 이렇다. 큐에서 대상 하나를 고른다. 원천 파일과 관련 파일을 읽는다. 타입 템플릿에 맞춰 노트를 쓴다. frontmatter의 `calls`, `reads`, `writes`와 본문 wikilink가 맞는지 확인한다. 모르는 것은 `unknowns`에 남기고, 해시와 분석 날짜를 기록한다.

이 방식은 느리다. 하지만 LLM 문서화에서 속도보다 중요한 것은 추적 가능성이다. "왜 이 노트가 이렇게 쓰였는가"를 나중에 확인할 수 없다면, 위키는 빠르게 믿기 어려운 요약 모음이 된다.

## 샌드박스 검증

실제 업무 환경에 넣기 전에 가짜 레거시 시스템 샌드박스를 만들었다. 화면, Java 계층, 쿼리, DB 오브젝트가 이어지는 작은 세계를 만들어 LLM이 템플릿을 잘 따르는지 확인했다.

검증은 해피 패스만 보지 않았다. 정상 케이스 7개와 악성 케이스 3개, 총 10개를 수동 시뮬레이션했다. 악성 케이스에는 소스 파일 없음, 동적 SQL, 오버로드 함수 같은 상황을 넣었다. 이런 경우 LLM은 자주 그럴듯한 추측을 한다. 그래서 "없으면 없다고 쓰기", "동적 SQL은 확정하지 않기", "오버로드는 식별 조건을 남기기" 같은 규칙이 필요했다.

Sonnet 서브에이전트 검증도 돌렸다. 첫 번째 검증은 서로 다른 세 객체를 병렬로 맡겨 핵심 필드가 기준 결과와 맞는지 봤고, 두 번째 검증은 화면부터 Java 3계층까지 이어지는 흐름을 확인했다. 여기서 날짜 추론 오류, forward-only 위반, Java 경로 규칙 불일치, 샘플 정합성 문제가 잡혔다.

`last_analyzed`는 특히 좋은 예다. LLM은 queue에 적힌 날짜를 분석 날짜처럼 추론하려 했다. 그래서 프롬프트를 바꿨다. 분석 날짜는 추론하지 말고 업무 환경에서 `date +%Y-%m-%d` 실행 결과를 쓰게 했다. 작은 차이처럼 보이지만, 나중에 위키를 운영할 때는 이런 필드 하나가 신뢰를 만든다.

## Graphify 없이도 확인할 방법

이 워크벤치도 그래프를 염두에 뒀다. 객체 간 `calls`, `reads`, `writes` 관계가 쌓이면 레거시 시스템의 지도가 된다. 하지만 Graphify 설치나 실행이 업무 환경에서 항상 가능하다고 가정할 수는 없다.

그래서 그래프 도구가 없어도 확인할 수 있는 fallback을 남겼다. `rg "calls|reads|writes"`로 frontmatter 관계를 훑고, `rg "\[\[.*\]\]"`로 본문 wikilink를 확인한다. 이상적인 도구가 막혔을 때도 최소한의 역방향 질의와 정합성 확인이 가능해야 한다.

이것은 제약 환경에서 자주 반복되는 교훈이다. 좋은 도구를 쓰는 것보다, 그 도구가 없을 때도 망가지지 않는 운영 경로가 더 중요하다.

## AI가 놓치기 쉬운 것

AI는 요약을 잘한다. 하지만 업무용 위키에서 요약 품질만으로는 부족하다. 어떤 파일을 보고 썼는지, 그 파일이 바뀌었는지, 어느 링크가 확실하고 어느 링크가 추정인지 남아야 한다.

또 AI는 기존 내용을 쉽게 다시 쓰려 한다. 해시가 같으면 확정 내용을 유지해야 하는데, 더 좋은 표현으로 바꾸고 싶어 한다. 이건 문장 품질로는 좋아 보여도 운영 기록으로는 위험하다. 위키는 에세이가 아니라 추적 가능한 지식 저장소다.

forward-only 규칙도 마찬가지였다. 아직 존재하지 않는 노트를 미리 확정 링크처럼 걸면 그래프가 빨리 풍성해 보인다. 하지만 나중에 실제 분석 결과와 어긋날 수 있다. 그래서 모르는 대상은 큐와 unknowns에 남기고, 확정 링크는 실제 노트가 생긴 뒤 연결하는 편이 낫다.

## 지금 상태

이 워크벤치는 실제 업무 환경의 첫 주기를 기다리는 단계였다. 하지만 집에서 할 수 있는 일은 충분히 했다. 타입을 정하고, 템플릿을 만들고, 프롬프트를 체크리스트화하고, 샌드박스를 만들고, 악성 케이스로 흔들어보고, 서브에이전트 검증으로 빈틈을 찾았다.

LLM 문서화의 핵심은 "잘 요약하기"가 아니라 "나중에 믿을 수 있게 남기기"다. `source_hash`, `unknowns`, 한 사이클 한 노트, frontmatter와 wikilink 정합성은 그 신뢰를 위한 작은 장치들이다. 느리지만 이 순서가 맞다.
