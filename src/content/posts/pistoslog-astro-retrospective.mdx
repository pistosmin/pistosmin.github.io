이번 블로그 작업은 화면을 예쁘게 바꾸는 일로 시작했지만, 실제로는 기록의 기준을 다시 세우는 일이었다. Claude로 진행했던 프로젝트와 Codex로 이관한 작업이 로컬 저장소, 세션 로그, 위키, 임시 문서에 흩어져 있었다. 결과물은 남아 있었지만 왜 만들었는지, 어디서 막혔는지, 어떤 기준으로 줄였는지는 바로 보이지 않았다.

그래서 `pistoslog`는 단순한 GitHub Pages 블로그가 아니라 작업 기록을 다시 꺼내 읽는 입구가 되어야 했다. 완성된 자랑만 올리는 곳이 아니라, 개발 날짜와 판단의 변화를 따라가며 다시 배울 수 있는 공간이 필요했다.

## 처음 선택한 구조

2026년 5월 5일 밤, 기존 정적 템플릿을 걷어내고 Astro 기반 블로그로 바꿨다. Astro를 고른 이유는 명확했다. 블로그의 shell, nav, footer, 글 카드, 상세 페이지는 대부분 정적 HTML이면 충분하다. React 앱처럼 전체 페이지를 hydration할 이유가 없다.

콘텐츠는 MDX와 Astro content collections로 관리했다. [Astro content collections 문서](https://docs.astro.build/en/guides/content-collections/)에서도 collection은 구조가 비슷한 콘텐츠 묶음을 관리하고, schema로 타입 체크를 붙이는 방식으로 설명한다. `posts`와 `projects`를 나누고 frontmatter schema를 강제했다. 글은 완성된 서사와 공부 기록을 담고, 프로젝트는 결과물과 맥락을 보여주는 자리다. 이 구분이 있어야 나중에 글이 늘어도 "이건 어디에 써야 하지"에서 덜 막힌다.

검색은 [Pagefind](https://pagefind.app/docs/)를 붙였다. Pagefind는 정적 사이트 생성기가 만든 HTML을 빌드 후 색인하고, 검색 번들을 생성된 사이트 안에 넣는다. GitHub Pages처럼 서버가 없는 정적 호스팅에서 검색을 제공하기 좋다. RSS와 sitemap은 블로그의 기본 배포 표면이다. 댓글과 반응은 [Giscus](https://giscus.vercel.app/)를 염두에 두었다. Giscus는 GitHub Discussions 기반 iframe이므로 public repo, 앱 설치, Discussions 활성화, repo/category id 같은 설정이 필요하다. 설정이 없을 때는 무리해서 실패하지 않고 "연동 대기" 상태를 보여주는 쪽으로 잡았다.

이 선택들은 전부 static-first라는 기준 안에 있었다. 문제는 그다음이었다. 시각 효과를 욕심내면서 이 기준이 흔들렸다.

## Refractive와 glass를 켰다 끄던 밤

처음에는 리퀴드 글래스, 굴절 효과, 움직이는 nav, 검색, 테마, 카드 메타데이터를 한꺼번에 얹었다. 효과는 재미있었다. 하지만 블로그는 효과가 아니라 읽기가 중심이어야 한다.

5월 5일 22시대 커밋을 보면 원인 분리 과정이 그대로 남아 있다. 먼저 Refractive runtime을 껐다. 그다음 Refractive는 살리고 CSS `backdrop-filter`와 큰 blur, 전역 pointer tracking을 제거해봤다. 다시 Refractive를 비활성화했다. 문제의 원인이 라이브러리인지, CSS blur인지, fixed background인지, React island인지 한 번에 확정할 수 없었기 때문이다.

이 과정에서 깨달은 것은 "예쁜 효과 하나가 문제"가 아니라 "블로그 shell이 interactive runtime에 기대고 있다"는 구조적 문제였다. nav와 header는 정적으로도 충분한데 React `client:only` island로 감싸고 있었다. 블로그의 기본 골격이 hydration과 런타임 효과에 의존하는 순간, 읽는 경험보다 효과의 안정성이 먼저 문제가 된다.

## static Astro shell로 돌아가기

전환점은 5월 5일 23시 6분 커밋이었다. Header를 React island에서 Astro 컴포넌트로 빼고, shell을 정적 구조로 다시 세웠다. 검색과 테마처럼 실제 상호작용이 필요한 부분만 작은 island로 남기는 방향이었다.

이 결정을 회귀 방지로 고정하기 위해 `scripts/verify-architecture.mjs`도 추가했다. 이 스크립트는 production 코드에 다음 패턴이 다시 들어오지 않는지 확인한다.

- 전체 Header를 React `client:only`로 만드는 것
- 기본 레이아웃에 `client:only`를 남기는 것
- 전역 `pointermove` 추적을 넣는 것
- production CSS에 큰 blur나 `backdrop-filter`를 되돌리는 것
- Refractive runtime dependency를 다시 production 의존성으로 넣는 것
- 예전 React Header import를 되살리는 것

이 스크립트는 단순한 취향 검사가 아니다. 한 번 겪은 아키텍처 회귀를 명령으로 막는 장치다. "앞으로 조심하자"보다 `npm run verify:architecture`가 더 오래 간다.

## 한국어 편집형 블로그로 방향을 바꾸다

기술 구조가 안정된 뒤에는 디자인 방향도 바꿨다. 처음의 visual 실험은 "AI가 만든 화려한 포트폴리오"처럼 보일 위험이 있었다. 그래서 5월 5일 23시 25분 이후에는 한국어 편집형 블로그에 맞게 정리했다.

Pretendard 기반의 읽기 좋은 글자, 종이 같은 표면, 낮은 장식 밀도, 명확한 카테고리와 태그, 모바일에서 줄이 깨지지 않는 본문 폭을 우선했다. 움직이는 nav glass capsule도 제거했다. hover와 active 상태는 정적으로 충분했다.

이 선택의 결론은 효과를 포기한 것이 아니었다. 효과가 없어도 읽히는 블로그를 먼저 만든 것이다. 좋은 장식은 본문을 밀어내지 않는다. 특히 한국어 글은 영문보다 줄바꿈과 호흡이 다르게 보이기 때문에, 카드 안의 작은 텍스트와 모바일 폭을 더 조심해야 한다.

## 홈 화면은 빈 카드가 아니어야 했다

5월 6일 0시대에는 홈 메타데이터를 다듬었다. 카테고리 요약, 최신 글 표시, NEW/updated 상태, 태그, 최근 반응, Giscus 연동 상태를 정리했다. 이 작업은 사소해 보이지만 중요했다.

AI가 만든 블로그 초안은 종종 "카드가 많지만 읽을 이유가 없는 화면"이 된다. 제목, 설명, 태그가 모두 비슷하고, 최신성이 보이지 않고, 어떤 글이 어떤 맥락인지 드러나지 않는다. 그래서 홈은 단순한 목록이 아니라 기록의 흐름을 보여줘야 했다.

이번에 만든 8편의 회고도 같은 기준을 적용했다. 개발한 날짜를 `published`에 반영하고, 하루에 한꺼번에 쓴 것처럼 보이지 않게 했다. 실제로 작업이 일어난 순서대로 gravity, 면접 준비 앱, querycreator, 오프라인 AI 루프, pistoswiki, 업무용 LLM 위키, DB 오브젝트 동기화, pistoslog가 이어지도록 했다.

## 글쓰기 기준도 다시 세웠다

처음 쓴 8편 초안은 너무 압축되어 있었다. "무엇을 만들었다"는 있었지만 "왜 그렇게 선택했는지", "어떤 검증을 했는지", "어디서 오래 걸렸는지", "AI가 무엇을 놓쳤는지"가 부족했다.

이 블로그의 글은 포트폴리오 카드가 아니라 회고여야 한다. 그래서 앞으로의 기준을 이렇게 잡았다.

첫째, 개발 날짜를 숨기지 않는다. 나중에 읽었을 때 어떤 작업이 어떤 시기에 이어졌는지 보여야 한다.

둘째, 기술 선택에는 이유를 붙인다. Astro, Pagefind, SQLite, Python, Graphify, JWT, Ralph 같은 단어만 나열하면 글이 아니라 키워드 목록이 된다.

셋째, 검증 단계를 적는다. `npm run check`, `npm run build`, `python -m pytest`, `py_compile`, Playwright 캡처 같은 명령은 글을 현실에 붙여준다.

넷째, 느렸던 지점을 숨기지 않는다. 버전 충돌, print layout, search route, Graphify 한계, `tee` exit code, `source_hash` 검증 같은 장면이 있어야 글이 살아난다.

다섯째, AI의 한계를 적는다. AI가 빠르게 구조를 만들고 초안을 넓혀준 것은 사실이다. 동시에 과장된 문장, 불필요한 기능 확장, 공개 경계 누락, 실제 사용 맥락 오해도 반복됐다. 사람이 어떤 기준으로 줄였는지가 기록의 핵심이다.

## 검증

이번 블로그 구조에서 반복한 검증은 단순했다.

```bash
cd /Users/pistosmin/develop/pistoslog && npm run check
cd /Users/pistosmin/develop/pistoslog && npm run build
cd /Users/pistosmin/develop/pistoslog && npm run verify:architecture
cd /Users/pistosmin/develop/pistoslog && git diff --check
```

현재 `package.json` 기준으로 `npm run build`는 `astro check && astro build && pagefind --site dist`를 함께 실행한다. 그래서 타입 체크, 정적 빌드, Pagefind 인덱스 생성을 한 번에 확인할 수 있다. GitHub Pages에 올릴 때는 base path와 `/pagefind/` asset 경로가 맞는지도 같이 봐야 한다.

시각 검증은 홈과 글 목록, 모바일 폭을 중심으로 봤다. 특히 390px 정도의 모바일 폭에서 텍스트가 카드 밖으로 밀리거나, nav와 버튼이 겹치거나, 긴 한국어 문장이 어색하게 잘리는지 확인했다.

## 지금의 결론

`pistoslog`를 다시 만든 일은 블로그 리디자인이기도 하지만, 더 정확히는 기록 운영 방식을 정한 일이다. 위키에는 원천을 남기고, 블로그에는 공개 가능한 서사를 쓴다. 날짜는 개발 시점에 맞추고, 글은 결과보다 판단의 변화를 보여준다.

예쁜 효과는 나중에 다시 실험할 수 있다. 하지만 효과가 없어도 읽히는 구조, 검증 가능한 아키텍처, 공개 가능한 글쓰기 기준은 먼저 필요하다. 이번 작업의 결론은 거기에 있다.
