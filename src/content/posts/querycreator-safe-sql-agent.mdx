`querycreator`는 자연어 질문을 안전한 RDB 조회로 바꾸는 도구다. 현업에서 "이 주문 상태가 왜 이렇지?", "어제 특정 설비의 처리량이 얼마였지?", "이 코드값은 어떤 의미지?" 같은 질문이 반복될 때마다 담당자가 직접 테이블을 찾고 SQL을 작성하는 흐름을 줄이기 위해 시작했다.

중요한 것은 "LLM이 SQL을 생성한다"는 멋진 문장이 아니었다. 운영 데이터베이스 앞에서 LLM을 어떻게 멈추게 할 것인가였다. LLM이 만든 SQL은 그럴듯해 보일 수 있지만, 운영 DB에서는 그럴듯함이 안전을 보장하지 않는다.

## 챗봇보다 도구 레이어가 먼저였다

처음부터 별도 웹 챗봇 UI를 만들 생각은 없었다. 이미 사내에서 LLM을 호출할 수 있는 경로가 있었고, 내가 만들고 싶었던 것은 그 LLM이 사용할 수 있는 안전한 DB 도구였다. 그래서 인터페이스를 크게 벌리지 않고 `get_metadata`, `execute_query`, `call_function` 세 가지로 좁혔다.

`get_metadata`는 질문과 관련된 업무 사전, 테이블, 컬럼, 인덱스, 제약조건, 공통코드 정보를 제공한다. `execute_query`는 검증된 SELECT만 실행한다. `call_function`은 허용 목록에 오른 저장 함수만 제한적으로 호출한다. 이 구조는 특정 UI에 묶이지 않고, 나중에 다른 Agent 플랫폼이나 CLI로 옮기기 쉬운 형태였다.

Python을 고른 이유도 현실적이었다. DB-API 계열 연결을 재사용하기 좋고, `oracledb`, `PyYAML`, `sqlparse`, `pytest` 같은 도구로 메타데이터 수집, YAML 업무 사전, SQL 파싱, 테스트를 한 흐름으로 묶기 쉽다. `sqlplus`와 spool 파일을 조합하면 빠르게 만들 수는 있지만, 텍스트 파싱과 환경 의존성이 커진다.

## 자연어 SQL보다 먼저 필요한 것

LLM에게 스키마 전체를 던져주면 답이 잘 나올 것 같지만 실제로는 그렇지 않다. 테이블이 많아질수록 토큰은 커지고, 중요한 업무 의미는 흐려진다. 컬럼명이 비슷하거나 약어가 많으면 LLM은 더 자신 있게 틀린다.

그래서 메타데이터를 네 겹으로 나눴다.

- 운영자가 쓰는 업무 사전
- DB에서 자동 수집하는 물리 메타데이터
- 공통코드와 코드값 설명
- 운영 중 발견한 지식 힌트

물리 메타데이터는 `all_tables`, `all_tab_columns`, `all_ind_columns`, `all_constraints`, `all_col_comments` 같은 시스템 뷰에서 읽었다. 사람이 모든 테이블을 손으로 설명하지 않아도 기본 구조는 자동으로 따라오게 하기 위해서다. 반대로 "이 테이블은 대용량이니 반드시 기간 조건이 필요하다", "이 컬럼은 실제 업무에서는 다른 이름으로 불린다" 같은 정보는 사람이 YAML로 보강했다.

이 분리는 꽤 중요했다. 자동 수집 가능한 사실과 사람이 아는 업무 맥락을 섞어버리면 둘 다 믿기 어려워진다. DB가 말해주는 구조는 DB에서 가져오고, 사람이 아는 위험한 조인 패턴이나 용어 차이는 별도 사전에 둔다.

## 안전 규칙은 기능이 아니라 제품의 중심이었다

가장 먼저 막은 것은 파괴적인 구문이었다. 조회 도구라면 기본은 SELECT여야 한다. 다중 구문도 막았다. 세미콜론 뒤에 다른 명령을 붙이는 순간 도구의 성격이 달라진다.

그다음은 성능과 오남용이었다. 일반 조회의 `SELECT *`는 금지했고, 허용 목록 함수 호출을 위한 `TABLE(...)` 패턴만 예외로 뒀다. 앞자리 wildcard LIKE도 제한했다. 대용량 테이블은 WHERE 조건 없이 실행하지 못하게 했다. 행 수 상한과 타임아웃도 넣었다. 허용되지 않은 함수 호출은 실행하지 않는다.

이 규칙들은 "LLM이 나쁜 의도를 가질 수 있다"는 전제만을 위한 것이 아니다. 오히려 더 흔한 위험은 LLM이 좋은 의도로 비싼 쿼리를 만드는 것이다. 사용자는 "어제 데이터 좀 보여줘"라고 물었는데, LLM은 기간 조건을 빠뜨리고 큰 테이블을 훑을 수 있다. 안전 레이어는 악의보다 확신에 찬 실수를 막기 위해 필요했다.

## 구현은 테스트 가능한 단위로 쪼갰다

커밋 흐름도 그 순서를 따른다. 2026년 4월 9일에는 설계 문서와 구현 계획을 만들고, 4월 10일에는 스캐폴딩, DB 추상화, 업무 사전, 메타데이터 카탈로그, SQL 검증기, 실행기, 도구 인터페이스, 로그 분석, E2E 테스트, 관리자 CLI, 문서를 차례로 붙였다.

초반에 Mock DB를 둔 것이 도움이 됐다. 실제 운영 DB에 붙어야만 테스트할 수 있는 구조였다면 개발 속도도 느리고 위험도 컸을 것이다. SQL validator, executor, metadata catalog, tool layer를 테스트로 나눠두면 실제 연결 전에도 대부분의 규칙을 확인할 수 있다.

최종 검증 기준은 다음 명령이었다.

```bash
cd /path/to/querycreator && python -m pytest tests/ -v --tb=short --cov=querycreator --cov-report=term-missing
```

당시 세션 기록 기준 최종 결과는 82개 테스트 통과와 89% 수준의 coverage였다. `test_validator.py`, `test_executor.py`, `test_tools.py`, `test_e2e.py`, `test_cli.py`처럼 기능별로 테스트가 나뉘었다. 여기서 중요한 것은 숫자 자체보다 테스트의 위치다. LLM이 만든 SQL을 실행하기 직전, 사람이 직접 눈으로 보기 어려운 규칙을 자동화된 검증으로 세워둔 것이다.

## 느렸던 지점

처음에는 기능 구현보다 패키징에서 시간이 나갔다. editable install과 빌드 백엔드가 흔들렸고, 시스템 Python 버전도 맞지 않았다. 결국 `setuptools.build_meta`와 pyenv Python 3.11 흐름으로 정리했다. 이런 문제는 멋진 설계와 상관없이 프로젝트를 바로 멈춘다.

범위도 계속 넓어졌다. 처음에는 SQL 생성과 검증이면 충분할 것 같았지만, 운영에 붙이려면 로그, 느린 쿼리 분석, 관리자 CLI, 온보딩 문서, API 문서가 필요했다. AI는 이 확장을 좋아한다. "이것도 있으면 좋다"는 목록을 계속 만든다. 하지만 첫 위험은 기능 부족이 아니라 잘못된 쿼리였기 때문에, 중심은 validator와 메타데이터 품질에 둬야 했다.

또 하나는 공개화였다. 내부 플랫폼명, 회사명, 실제 업무명은 공개 저장소에 그대로 남기면 안 된다. 그래서 공개 전에는 이름을 일반화하고, 원본 맥락은 보존하되 식별 가능한 표현은 걷어냈다. 기능 코드만 안전하면 공개 가능한 것이 아니라, README와 예시, 테스트 데이터, 커밋에 남은 이름까지 봐야 한다는 것을 다시 확인했다.

## 권한과 감사는 아직 남았다

아직 어려운 부분은 권한과 감사다. SELECT 전용 계정으로 시작하면 빠르지만, 사용자별 권한 분리와 감사 로그는 별도 설계가 필요하다. "누가 어떤 질문으로 어떤 SQL을 실행했는가"가 남아야 운영 도구가 된다.

그래도 `querycreator`는 이후 LLM 도구를 볼 때 기준이 됐다. LLM에게 무엇을 시킬지보다, LLM이 틀렸을 때 어느 레이어에서 멈출지 먼저 설계해야 한다. 자연어 SQL의 핵심은 생성 품질이 아니라 제한, 검증, 추적 가능성이다.
