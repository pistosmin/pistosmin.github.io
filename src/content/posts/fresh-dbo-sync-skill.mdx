업무용 LLM 위키를 만들 때 자바 소스만 읽어서는 부족하다. 실제 업무 로직은 DB 안에도 많이 있다. 프로시저, 함수, 패키지, 테이블, 뷰, 동의어, DB 링크 같은 오브젝트가 바뀌면 위키가 보고 있는 근거도 같이 바뀐다.

`fresh-dbo`는 이 문제를 풀기 위한 작은 스킬이었다. DB에 접속해 변경된 오브젝트만 로컬 SQL 파일로 가져오고, LLM 위키가 그 파일을 원천으로 다시 분석할 수 있게 만든다. 화려한 요약 도구가 아니라 원천을 신선하게 유지하는 동기화 레이어다.

## DB에는 쓰지 않기로 했다

DB에서 로컬 파일로 가져오기만 하는 선을 먼저 그었다. 로컬 파일을 다시 DB에 반영하는 기능은 넣지 않는다. 이 프로젝트의 목적은 배포나 변경 관리가 아니라 분석과 추적이다.

두 번째 기준은 기존 연결 방식을 재사용하는 것이다. 제한된 업무 환경에서는 이미 검증된 접속 함수와 계정 정책이 있다. AI는 처음에 접속 방식부터 새로 설계하려 했다. 하지만 새 라이브러리 설치, 새로운 접속 문자열, 별도 계정 생성은 모두 운영 제약에 걸릴 수 있다. 그래서 이 스킬은 변경 감지와 파일 갱신 흐름만 책임지고, 실제 연결은 기존 함수로 갈아끼울 수 있게 잡았다.

## JSON에서 SQLite로 바뀐 순간

초기 제안은 단순했다. `.fetch_state.json` 같은 파일에 마지막 동기화 상태를 저장하자는 방식이었다. 작은 스키마라면 충분해 보인다. 하지만 사용자가 바로 지적했다. "오브젝트별로 관리하는 게 아니냐", "json이 수천수만 줄이면 어떻게 하냐"는 질문이었다.

그 질문이 설계를 바꿨다. DB 오브젝트는 스키마 단위 한 줄이 아니라 오브젝트 단위로 추적해야 한다. 신규, 변경, 삭제, 실패 재시도, run log를 생각하면 JSON 한 파일은 금방 다루기 어려워진다. 그래서 Python 표준 `sqlite3`를 써서 로컬 상태 파일을 두는 구조로 바꿨다.

상태 테이블은 크게 세 가지다.

- `objects`: 현재 추적 중인 오브젝트와 마지막 DDL 시간, fetch 시각, 파일 경로
- `deleted_objects`: DB에서 사라진 오브젝트의 삭제 기록
- `run_log`: 실행 시각, 성공/실패, 통계, 오류 메시지

이 변경은 작지만 중요했다. LLM 위키에서 신뢰할 수 있는 것은 "마지막으로 언제 봤는가"와 "무엇이 바뀌었는가"다. 상태 저장소가 대충이면 위키의 재분석 큐도 대충이 된다.

## 변경 감지 흐름

기본 흐름은 단순하다. `USER_OBJECTS`에서 `object_type`, `object_name`, `last_ddl_time`을 읽는다. 이전 SQLite 상태와 비교한다. 처음 보는 오브젝트는 가져오고, `last_ddl_time`이 달라진 오브젝트는 다시 가져온다. DB 목록에는 없는데 상태에 남아 있는 오브젝트는 삭제된 것으로 판단해 로컬 파일을 정리하고 `DELETED.md` 같은 기록에 남긴다.

DDL 추출은 `DBMS_METADATA.GET_DDL`을 우선 생각했다. 테이블이나 뷰처럼 DDL 형태로 가져오는 것이 자연스러운 오브젝트가 많기 때문이다. 다만 모든 환경에서 동일하게 잘 되리라고 가정하지는 않았다. 실패하거나 소스 기반으로 읽어야 하는 경우에는 `USER_SOURCE` 계열로 fallback할 수 있게 했다.

파일은 타입별 폴더에 저장한다. 사람이 훑어볼 때도 `TABLE`, `VIEW`, `PROCEDURE`, `FUNCTION`, `PACKAGE_BODY`, `DB_LINK`처럼 나뉘어 있으면 좋다. LLM도 경로만 보고 오브젝트의 성격을 추론하기 쉬워진다.

## 검증은 어디까지 했나

이 작업은 실제 운영 DB에 직접 붙여 끝까지 돌린 단계는 아니었다. 그래서 검증 범위를 정직하게 나눴다. 실제 DB 연결이 필요한 부분은 업무 환경에서 교체해야 하고, 로컬에서는 CLI parser, import, 컴파일, 도움말, 패키징을 확인했다.

```bash
cd /path/to/fresh-dbo && python3 -c "import fetch_db_objects.cli; fetch_db_objects.cli.build_parser().parse_args(['--db', 'oracle', '--schema', 'HR', '--local-root', '/tmp/x'])"
cd /path/to/fresh-dbo && python3 -m py_compile fetch_db_objects/*.py
cd /path/to/fresh-dbo && python -m fetch_db_objects --help
```

패키징은 zip으로 만들고, 전달 포맷 제약에 맞춰 확장자를 조정한 뒤 `file`과 `unzip -l`로 내용이 정상인지 확인했다. 여기서도 AI가 한 번 과하게 해석했다. 사용자가 말한 `.img`는 디스크 이미지나 zipapp을 만들라는 뜻이 아니라, 전달 포맷 제약에 맞춘 확장자 변경에 가까웠다. AI는 형식 가능성을 넓게 보지만, 실제 의도는 더 단순할 때가 많다.

## 오래 걸린 이유

기술적으로 어려운 부분은 DDL 추출보다 경계 설정이었다. 어디까지 이 스킬이 책임지고, 어디부터 업무 환경의 기존 연결 함수에 맡길 것인가. 어떤 오브젝트를 가져오고, 실패한 오브젝트는 전체 실행을 멈추게 할 것인가. 삭제된 오브젝트는 파일을 그냥 지울 것인가, 별도 기록을 남길 것인가.

처음에는 새 접속 계층과 이상적인 RDB 추상화를 만들고 싶어졌다. 하지만 제약 환경에서는 이미 되는 연결을 재사용하는 것이 가장 안전하다. 이 스킬은 DB 접속 프레임워크가 아니라 변경 감지기다. 그 범위를 줄인 뒤 구조가 선명해졌다.

RDB별 차이도 남아 있다. DDL 포맷, 대용량 source 처리, 권한, 패키지 body 접근, DB link 노출 여부는 환경마다 다르다. 그래서 실패한 오브젝트는 전체 실행을 망치지 않고 다음 실행에서 재시도할 수 있어야 한다.

## LLM 위키에서 왜 중요한가

LLM 위키는 원천이 오래되면 바로 무너진다. 노트가 아무리 잘 쓰여 있어도, 그 노트가 본 프로시저가 이미 바뀌었다면 신뢰할 수 없다. `fresh-dbo`가 하는 일은 요약이 아니라 재분석 신호를 만드는 것이다.

어떤 오브젝트가 언제 바뀌었고, 로컬 파일이 언제 갱신됐고, 어떤 파일을 보고 위키 노트가 작성됐는지 이어져야 한다. 그래야 `source_hash`와 `last_analyzed` 같은 필드가 실제 의미를 갖는다.

이 프로젝트는 작은 도구지만 기준은 분명했다. 레거시를 바꾸지 않고, LLM이 읽을 수 있는 표면을 만든다. 그리고 그 표면이 오래되지 않게 변경 감지를 붙인다. 문서화 자동화의 숨은 핵심은 좋은 문장보다 신선한 원천이다.
