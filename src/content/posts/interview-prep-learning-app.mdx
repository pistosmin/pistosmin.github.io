처음부터 웹앱을 만들 생각은 아니었다. 목표는 훨씬 현실적이었다. 경력 지원 마감이 가까웠고, 내가 해온 일을 지원 직무의 언어로 다시 정리해야 했다. 이력서를 고치고, 기술 면접에서 받을 질문을 예상하고, 실제로 말해볼 답변을 준비하면 되는 일이었다.

그런데 2026년 3월 28일 오후부터 작업은 점점 앱이 됐다. 이력서 탭, 기술 스터디 탭, 퀴즈 탭, 면접 탭이 생겼다. PDF 출력이 붙고, 검색과 용어사전이 붙고, 테마와 대시보드가 붙었다. 며칠 뒤에는 여러 버전의 이력서를 관리하는 구조까지 들어갔다.

겉으로 보면 생산적인 작업처럼 보인다. 하지만 이 프로젝트의 핵심 회고는 조금 불편하다. 도구를 만드는 일이 실제 면접 준비를 밀어냈다.

## 내가 풀고 싶었던 진짜 문제

문제는 이력서 앱이 아니었다. 내가 해온 일을 어떻게 말할 것인가였다.

레거시 업무 시스템에서 오래 일하면 경험은 많아진다. 장애를 보고, 배치를 보고, 쿼리를 보고, 오래된 프레임워크의 제약 속에서 운영을 배운다. 그런데 그 경험을 그대로 적으면 "낡은 시스템을 유지보수한 사람"처럼 보일 수 있다. 반대로 너무 예쁘게 포장하면 면접에서 설명할 수 없는 말이 된다.

그래서 초반에 가장 많이 세운 기준은 정직성이었다. 실제로 직접 구현한 것과 학습해서 설명 가능한 것을 분리한다. 업무 도메인의 세부 표현보다 기술적 문제 해결을 앞에 둔다. 시스템 규모나 수치를 과장하지 않는다. 면접에서 질문이 들어왔을 때 내 말로 설명할 수 있는 문장만 이력서에 남긴다.

이 기준은 AI가 자동으로 세워주지 못했다. AI는 경력 내용을 더 멋있게 만들려 했다. 규모를 앞세우거나, 특정 시스템의 내부 표현을 많이 넣거나, 직접 하지 않은 기술까지 자연스럽게 섞으려 했다. 그때마다 내가 줄이고, 이름을 바꾸고, 실제로 말할 수 있는 선을 다시 그었다.

## 앱이 된 이유

처음에는 문서로 충분할 것 같았다. 그런데 기술 딥다이브를 하다 보니 주제가 늘어났다. 메시지 큐, 분산 트랜잭션, 레거시 전환, 검색, LLM 도구, 데이터베이스 격리수준 같은 주제를 정리해야 했다. 각 주제는 정의, 원리, 코드 예시, 다이어그램, 면접 답변까지 필요했다.

여기서 AI는 강했다. 질문 목록을 빠르게 만들고, 주제별 설명을 뽑고, 학습 순서를 정리했다. 하지만 양이 늘면서 문서가 읽기 어려워졌다. 그래서 Vite, React, TypeScript로 로컬 웹앱을 만들었다. 이력서와 스터디를 탭으로 나누고, 퀴즈로 반복 학습하고, `react-to-print`로 출력까지 하자는 생각이었다.

기술 선택 자체는 이상하지 않았다. 빠른 로컬 확인에는 Vite가 좋고, 화면 상태가 많은 학습 도구에는 React가 편했다. 코드와 다이어그램을 보여주려면 Prism.js와 Mermaid가 유용했다. 사용자가 "요즘 npm보다 pnpm을 많이 쓴다"고 지적한 뒤에는 패키지 매니저도 pnpm으로 바꿨다.

문제는 선택이 아니라 타이밍이었다. D-day가 있는 작업에서 앱의 완성도는 학습의 대체물이 될 수 없다.

## 검증한 것들

앱이 된 이상 최소한 깨지지 않아야 했다. 빌드와 린트, 로컬 서버 실행을 반복했다.

```bash
cd /path/to/interview-prep/web && pnpm install
cd /path/to/interview-prep/web && pnpm run build
cd /path/to/interview-prep/web && pnpm run lint
cd /path/to/interview-prep/web && pnpm run dev
```

화면 검증도 했다. 대시보드, 이력서, 검색 결과, 프린트 페이지 스크린샷을 남겼고, Playwright로 몇몇 흐름을 확인했다. 특히 PDF 출력은 웹 화면과 결과물이 달라서 따로 봐야 했다. 브라우저에서는 예뻐 보여도 인쇄 페이지에서는 여백, 줄바꿈, 색 대비, 페이지 나눔이 전부 달라진다.

## 가장 오래 걸린 부분

가장 길게 끌린 것은 검색과 용어사전, 뒤로가기, 스크롤 복원, 프린트 레이아웃이었다. 검색어가 남아 있어 다른 화면에서 이상하게 보이거나, 용어 링크를 눌렀는데 문서 상단으로 가거나, glossary route에서 아무것도 안 나오는 일이 반복됐다. 뒤로가기를 하면 원래 읽던 위치로 돌아와야 하는데 스크롤이 깨지는 문제도 있었다.

테마도 기대보다 오래 걸렸다. "글래스모피즘" 같은 단어를 주면 AI는 그럴듯한 연보라 배경이나 반투명 카드부터 만든다. 하지만 면접 준비 도구에서 중요한 건 장식보다 빠르게 읽히는 밀도다. 다크모드에서는 대비가 부족했고, 프린트에서는 색이 방해됐다. 결국 테마도 사용 목적에 맞게 다시 줄여야 했다.

이력서 표현도 계속 고쳤다. AI는 시스템 규모, WAS 대수, 처리량, 도메인 표현을 앞에 세우려 했다. 하지만 좋은 개발자 이력서는 "내가 어떤 문제를 어떤 판단으로 풀었는가"가 먼저다. 숫자는 근거가 확실할 때만 들어가야 하고, 도메인 세부 표현은 면접관이 관심을 가질 정도로만 남겨야 한다.

## AI가 도와준 것과 놓친 것

AI는 자료를 늘리는 데 탁월했다. 기술 질문 100개를 만들고, 주제별 설명을 초안으로 뽑고, 화면 구조를 빠르게 제안했다. 혼자 했다면 며칠 걸릴 양을 몇 시간 안에 펼쳐놓을 수 있었다.

하지만 AI는 "지금 이 순간 무엇을 하지 말아야 하는가"에 약했다. 마감 직전에는 좋은 앱보다 닫힌 이력서 한 장이 먼저다. 질문 100개보다 실제로 말해본 답변 10개가 더 중요하다. 그날 밤에는 앱을 더 고치는 대신, 화면을 닫고 답변을 소리 내어 말해보는 쪽을 택했어야 했다.

가장 중요한 교정은 "아는 척하지 말라"였다. 직접 구현하지 않은 기술은 학습 주제로 두고, 실제 경험처럼 쓰지 않는다. JD의 필수 스킬을 "학습 중"이라고 전면에 드러내는 것도 좋지 않았다. 부족한 것을 숨기라는 뜻이 아니라, 이력서의 첫 화면은 약점 목록이 아니라 문제 해결 경험이어야 했다.

## 이 프로젝트가 남긴 결론

도구 만들기는 학습처럼 느껴진다. 특히 개발자에게는 더 그렇다. 화면이 생기고, 검색이 되고, PDF가 나오면 뭔가 진전한 것 같다. 하지만 면접 준비의 목표는 앱을 완성하는 것이 아니라 내 경험을 정확하게 말하는 것이다.

다시 한다면 순서를 바꾼다. 먼저 이력서 한 장을 끝낸다. 그다음 하루에 질문 10개씩 실제로 말한다. 기술 스터디는 정의, 원리, 코드, 그림, 면접 답변까지 한 주제씩 닫는다. 웹앱은 시간이 남으면 만든다.

이 프로젝트는 실패라기보다 경고에 가깝다. AI와 함께하면 무언가를 많이 만들 수 있다. 하지만 많이 만든 것이 곧 준비가 된 것은 아니다. 좋은 협업은 더 많이 만드는 능력이 아니라, 지금 필요 없는 것을 덜어내는 기준에서 시작한다.
