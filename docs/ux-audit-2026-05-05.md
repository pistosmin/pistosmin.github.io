# pistoslog UI/UX 점검

작성일: 2026-05-05

## 점검 기준

현재 코드 기준으로 글 작성, 글 탐색, 글 읽기, 댓글 남기기, 페이지 이동, 재방문 흐름을 확인했다. 로컬 브라우저 테스트는 제외하고 코드 구조와 정적 빌드 결과를 기준으로 정리한다.

## 요약

블로그의 방향성은 명확하다. 홈은 “오래 읽히는 개인 기록 공간”이라는 인상을 잘 주고, 글/노트/프로젝트가 분리되어 있어 콘텐츠 작성자 입장에서도 운영 단위가 이해하기 쉽다. 다만 실제 사용자가 여러 글을 오가며 읽는 흐름에서는 현재 위치 표시, 뒤로가기, 읽은 글 표시, 댓글 상태 안내가 아직 약하다. 검색과 댓글처럼 사용자의 행동을 받아주는 기능은 기본 뼈대는 있으나 접근성, 상태 표시, 배포 설정 확인이 필요하다.

## 우선순위 높은 개선 포인트

### 1. 현재 페이지 표시가 없다

- 위치: `src/components/Header.tsx`, `src/layouts/BaseLayout.astro`
- 문제: 상단 내비게이션이 모든 페이지에서 같은 스타일로만 보인다. 사용자는 `Writing`, `Notes`, `Projects`, `About`, `Now` 중 어디에 있는지 시각적으로 확인하기 어렵다.
- 영향: 목록에서 상세로 이동하거나 여러 페이지를 오갈 때 방향 감각이 약해진다.
- 개선안: 현재 경로와 `navItems`를 비교해 해당 링크에 `aria-current="page"`와 active 클래스를 추가한다. 스타일은 배경을 과하게 바꾸기보다 텍스트 색, 얇은 테두리, 작은 점 정도로 충분하다.

### 2. 상세 페이지의 뒤로가기가 사용자의 진입 경로를 보존하지 않는다

- 위치: `src/pages/posts/[...slug].astro`, `src/pages/notes/[...slug].astro`, `src/pages/projects/[...slug].astro`
- 문제: 상세 페이지의 `back-link`가 항상 `/writing/`, `/notes/`, `/projects/`로 간다. 사용자가 홈의 최근 글, 검색 결과, 관련 글에서 들어왔어도 원래 위치로 돌아가지 못한다.
- 영향: 탐색 흐름이 끊기고, 검색 결과에서 여러 글을 비교해 읽기 불편하다.
- 개선안: 기본 링크는 목록으로 두되, 클라이언트에서 `document.referrer`나 `history.length`를 확인해 같은 사이트 내부에서 온 경우 `history.back()`을 쓰는 버튼을 제공한다. 라벨은 `목록으로`보다 `이전 화면`과 `글 목록`을 분리하는 편이 명확하다.

### 3. 읽은 글/봤던 글 표시가 없다

- 위치: `src/components/PostCard.astro`, `src/pages/posts/[...slug].astro`, `src/styles/global.css`
- 문제: 사용자가 어떤 글을 이미 열어봤는지 알 수 있는 표시가 없다. 브라우저 기본 `:visited` 스타일도 카드 디자인 안에서 거의 드러나지 않는다.
- 영향: 글 목록이나 관련 글에서 같은 글을 반복해서 열기 쉽다.
- 개선안: 최소 개선은 `.post-card__link:visited h3`처럼 제목 색을 미세하게 낮추는 것이다. 더 나은 방식은 글 상세 진입 시 `localStorage`에 post id를 기록하고, 카드에 `읽음` 배지나 낮은 대비 상태를 적용하는 것이다.

### 4. 홈 카테고리 인덱스가 빈 카테고리로 이동할 수 있다

- 위치: `src/pages/index.astro`, `src/pages/writing/index.astro`
- 문제: 홈 카테고리 인덱스는 모든 카테고리를 보여주며 `일상 0 posts`도 `/writing/#life`로 연결한다. 하지만 글 목록에서는 글이 없는 카테고리 섹션을 렌더링하지 않아 해당 anchor가 없다.
- 영향: 사용자가 빈 카테고리를 누르면 기대한 위치나 안내 없이 글 목록 상단에 머문다.
- 개선안: 홈에서는 글 수가 0인 카테고리를 비활성화하거나 숨긴다. 또는 글 목록에 빈 카테고리 섹션과 “아직 글이 없습니다” 안내를 둔다.

### 5. 모바일 검색 버튼의 접근성 이름이 약해질 수 있다

- 위치: `src/components/SearchDialog.tsx`, `src/styles/global.css`
- 문제: React 검색 버튼에는 `aria-label`이 없고, 모바일 CSS에서 텍스트와 `kbd`를 숨긴다. 현재 아이콘은 `aria-hidden`이라 화면 크기와 CSS 상태에 따라 보조기술이 버튼 이름을 잃을 수 있다.
- 영향: 키보드/스크린리더 사용자에게 검색 버튼이 무엇인지 불명확해질 수 있다.
- 개선안: `SearchDialog`의 열기 버튼에 `aria-label="검색"`을 추가한다. 검색 input에도 placeholder 외에 `aria-label="검색어"`를 추가한다.

### 6. 검색 다이얼로그에 포커스 트랩과 키보드 결과 탐색이 없다

- 위치: `src/components/SearchDialog.tsx`
- 문제: 다이얼로그가 열릴 때 input focus는 이동하지만, Tab 포커스가 배경 페이지로 빠져나가는 것을 막지 않는다. 결과 목록도 위/아래 방향키 탐색이나 첫 결과 Enter 이동을 지원하지 않는다.
- 영향: 검색은 블로그의 핵심 탐색 기능인데, 키보드 사용자에게 완성도가 낮게 느껴진다.
- 개선안: 다이얼로그 열기 전 focus element를 저장하고 닫을 때 복원한다. Tab 순환을 막고, 결과가 있을 때 `aria-live` 메시지와 방향키 탐색을 지원한다.

### 7. 댓글 영역의 실제 사용 상태가 불분명하다

- 위치: `src/components/Giscus.astro`, `.env.example`
- 문제: giscus 설정이 없으면 placeholder가 보이지만, 방문자 입장에서는 댓글이 준비 중인지, GitHub 로그인이 필요한지, 댓글을 어디에 남기는지 알기 어렵다. `.env.example`의 `PUBLIC_GISCUS_REPO`는 `pistosmin/pistoslog`인데 현재 GitHub에서 해당 저장소가 확인되지 않는다.
- 영향: 댓글을 남기려는 사용자가 기능이 막힌 것인지 아직 연결 전인지 판단하기 어렵다.
- 개선안: placeholder 문구를 방문자 기준으로 바꾼다. 예: “댓글은 GitHub Discussions 연결 후 열립니다.” 활성화 후에는 GitHub 로그인이 필요하다는 짧은 안내와 로딩/실패 상태를 준비한다.

### 8. 수동 테마와 giscus 테마가 어긋날 수 있다

- 위치: `src/components/ThemeToggle.tsx`, `src/components/Giscus.astro`
- 문제: 사이트 테마는 `localStorage`로 수동 전환하지만, giscus는 `data-theme="preferred_color_scheme"`를 사용한다. 사용자가 사이트에서 다크/라이트를 바꿔도 댓글 iframe은 OS 선호 테마를 따라갈 수 있다.
- 영향: 댓글 영역만 다른 테마로 보여 전체 경험이 끊긴다.
- 개선안: 테마 변경 시 giscus iframe에 `postMessage`로 theme 변경을 전달한다. 초기 `data-theme`도 현재 `document.documentElement.dataset.theme`와 맞추는 방식이 좋다.

### 9. 댓글보다 광고 placeholder가 먼저 나온다

- 위치: `src/pages/posts/[...slug].astro`, `src/components/AdSlot.astro`
- 문제: 글 본문 뒤에 바로 `AdSlot`이 나오고 그 다음 댓글이 나온다. 광고 값이 없을 때도 `AD` placeholder가 먼저 보인다.
- 영향: 댓글을 남기려는 사용자의 흐름이 끊기고, 개인 블로그의 대화 공간보다 광고 슬롯이 먼저 강조된다.
- 개선안: 초기에는 댓글을 본문 바로 아래에 두고 광고는 관련 글 이후나 하단으로 내린다. 광고가 설정되지 않았을 때는 placeholder를 운영자 확인용으로만 숨기거나 낮은 대비로 처리한다.

## 중간 우선순위 개선 포인트

### 10. 글 상세 목차가 읽기 진행 상태를 알려주지 않는다

- 위치: `src/pages/posts/[...slug].astro`, `src/styles/global.css`
- 문제: 목차는 anchor 목록만 제공한다. 현재 읽는 섹션, 진행률, 맨 위로 이동 같은 읽기 보조 기능은 없다.
- 영향: 긴 글이 늘어나면 사용자가 글 안에서 길을 잃기 쉽다.
- 개선안: IntersectionObserver로 현재 heading을 표시하고, 모바일에서는 접을 수 있는 `목차` 버튼 형태로 만든다. 글 상단이나 사이드에 읽기 진행률을 얇게 보여주는 것도 좋다.

### 11. 글 작성 스크립트가 한글 제목에서 slug 충돌을 만들 수 있다

- 위치: `scripts/new-content.mjs`
- 문제: 한글 제목은 ASCII 변환 후 빈 slug가 되어 `post-YYYYMMDD` 같은 fallback을 쓴다. 같은 날 한글 제목 글을 두 개 만들면 충돌한다.
- 영향: 작성자가 새 글을 만들다가 파일명 충돌을 자주 만날 수 있다.
- 개선안: `--slug` 옵션을 받거나, 한글 제목은 날짜 뒤에 짧은 해시를 붙인다. 실패 메시지도 “같은 날짜의 한글 제목은 slug를 지정해 주세요”처럼 다음 행동을 알려주는 편이 좋다.

### 12. 검색 기능이 빌드 후에만 활성화되는 점이 사용자에게는 자연스럽지만 운영자에게는 헷갈릴 수 있다

- 위치: `src/components/SearchDialog.tsx`, `package.json`
- 문제: dev에서는 “검색 인덱스는 프로덕션 빌드 후 활성화됩니다”라고 나온다. 방문자에게는 문제가 없지만, 운영자가 새 글을 쓴 뒤 검색 반영 여부를 확인하는 흐름이 README에 충분히 연결되어 있지 않다.
- 영향: 글 작성 후 검색 인덱스 확인을 놓치기 쉽다.
- 개선안: README의 작성 플로우에 `npm run build` 또는 배포 후 검색 확인 절차를 추가한다. 검색 다이얼로그 메시지는 운영자용 문구와 방문자용 문구를 분리해도 좋다.

### 13. 프로젝트 외부 링크가 같은 탭에서 열린다

- 위치: `src/pages/projects/[...slug].astro`, `src/components/RefractiveActionLinks.tsx`
- 문제: 프로젝트 상세의 repo/demo 링크가 외부 URL이어도 같은 탭에서 열린다.
- 영향: GitHub로 나간 뒤 블로그로 돌아오는 흐름이 약해진다.
- 개선안: 외부 URL은 `target="_blank" rel="noreferrer"`를 적용하고, 라벨을 `GitHub`, `Demo`, `Case study`처럼 방문자가 이해하기 쉬운 이름으로 매핑한다.

### 14. 업데이트 날짜가 스키마에는 있지만 화면에 표시되지 않는다

- 위치: `src/content.config.ts`, `src/pages/posts/[...slug].astro`, `src/pages/notes/[...slug].astro`, `src/pages/projects/[...slug].astro`
- 문제: 콘텐츠 스키마에는 `updated`가 있지만 상세 화면에는 발행일만 보인다.
- 영향: 기술 글이나 프로젝트 기록에서 정보가 최신인지 판단하기 어렵다.
- 개선안: `updated`가 있으면 “수정: 2026년 5월 5일”을 meta 영역에 작게 표시한다.

### 15. 모바일 내비게이션이 가로 스크롤임을 알기 어렵다

- 위치: `src/styles/global.css`
- 문제: 980px 이하에서 nav links는 가로 스크롤이지만 scrollbar가 숨겨져 있다.
- 영향: 좁은 화면에서 뒤쪽 메뉴가 보이지 않을 경우 추가 메뉴가 있다는 단서가 약하다.
- 개선안: 양 끝 fade mask, 마지막 메뉴 살짝 잘림, 또는 더보기 메뉴 중 하나를 적용한다. 현재 메뉴 수가 적다면 모바일에서는 2줄 래핑도 고려할 수 있다.

## 낮은 우선순위 개선 포인트

### 16. 카드 hover만 있고 focus 상태의 카드 전체감이 약하다

- 위치: `src/components/PostCard.astro`, `src/components/ProjectCard.astro`, `src/styles/global.css`
- 문제: 카드 hover는 분명하지만 키보드 focus 시 카드 전체의 상태 변화가 hover만큼 명확하지 않을 수 있다.
- 개선안: `.post-card:focus-within`, `.project-card:focus-within`에 hover와 비슷한 border/shadow를 준다.

### 17. 글 목록 필터가 anchor 이동만 제공한다

- 위치: `src/pages/writing/index.astro`
- 문제: 카테고리 칩을 누르면 필터링이 아니라 페이지 내 이동만 된다.
- 개선안: 글이 많아지면 `전체`, `일상`, `생각`, `기술공부`, `작업기록` 필터를 클라이언트에서 제공하거나, 카테고리별 별도 URL을 만든다.

### 18. related 영역이 글의 다음 행동을 충분히 제안하지 않는다

- 위치: `src/pages/posts/[...slug].astro`
- 문제: 관련 글은 같은 카테고리 2개만 보여준다. 글 수가 적거나 같은 카테고리가 없으면 다음 행동이 사라진다.
- 개선안: related가 없을 때 `최신 글`, `글 목록`, `Notes` 중 하나를 fallback으로 제공한다.

## 배포 확인 메모

- `npm run build`는 통과했다.
- 현재 로컬 git에는 아직 커밋이 없고 원격도 없다.
- `pistosmin/pistoslog` 저장소는 GitHub에서 확인되지 않았다.
- `pistosmin.github.io` 저장소는 이미 존재하며 `main` 루트가 GitHub Pages로 배포 중이다. 이 저장소에 현재 프로젝트를 올리면 기존 공개 사이트 파일을 교체하게 된다.

## 추천 작업 순서

1. 배포 대상 저장소를 확정한다.
2. `aria-current`, 검색 버튼/input 라벨, `:focus-within`처럼 작은 접근성 개선을 먼저 처리한다.
3. 홈의 0개 카테고리 링크와 상세 뒤로가기 흐름을 고친다.
4. 읽은 글 표시와 목차 active 상태를 추가한다.
5. giscus repo, theme sync, 댓글/광고 순서를 정리한다.
6. 글 작성 스크립트의 한글 slug 충돌을 해결한다.
